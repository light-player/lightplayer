# M1.1 — LPIR + Format Prerequisites

## Goal

Land the small, self-contained changes that M2.0 (synthetic
`__render_texture`) depends on. Bundled as one plan since each piece is
small and they're naturally co-developed.

## Scope

### `Store16` LPIR op

Add a new operation to write a 16-bit value to a base+offset address.
Required for all unorm16 texture formats — the existing `Store` is 32-bit.

```rust
LpirOp::Store16 {
    base: VReg,
    offset: u32,
    value: VReg,  // i32 source, lower 16 bits stored
}
```

Backend implementations:

- **Cranelift** (`lpvm-cranelift`): `MemFlags::trusted()` `istore16`
- **Native JIT** (`lpvm-native/rt_jit`): RV32 `sh` instruction
- **Emulator** (`lpvm-native/rt_emu`): mask + write u16
- **WASM** (`lpvm-wasm`): `i32.store16`

Lowering tests: round-trip a few values through each backend, verify
correct truncation and offset behavior.

### Additional `TextureStorageFormat` variants

Add two new formats alongside the existing `Rgba16Unorm`:

```rust
pub enum TextureStorageFormat {
    Rgba16Unorm,  // existing: 4 channels, 8 bpp
    Rgb16Unorm,   // NEW: 3 channels, 6 bpp
    R16Unorm,     // NEW: 1 channel, 2 bpp
}
```

Update `bytes_per_pixel()` and `channel_count()` for each variant.

### `compile_px` validation for new formats

Update `expected_return_type` in `lp-shader/src/engine.rs`:

```rust
fn expected_return_type(format: TextureStorageFormat) -> LpsType {
    match format {
        TextureStorageFormat::R16Unorm => LpsType::Float,
        TextureStorageFormat::Rgb16Unorm => LpsType::Vec3,
        TextureStorageFormat::Rgba16Unorm => LpsType::Vec4,
    }
}
```

Tests:
- `compile_px(R16Unorm)` accepts `float render(vec2)`, rejects `vec4`
- `compile_px(Rgb16Unorm)` accepts `vec3 render(vec2)`, rejects `vec4`
- `compile_px(Rgba16Unorm)` accepts `vec4` (existing)

### LPIR programmatic-construction spike

Investigate whether assembling a synthetic function (`__render_texture`)
via direct `LpirOp` pushes is ergonomic enough, or whether we need a small
builder helper.

Output: either a minimal builder helper module, or a "we have what we
need" note documenting the chosen pattern with a small example.

This is exploratory and may add zero new code — its purpose is to
de-risk M2.0's LPIR synthesis step.

## Validation

```bash
cargo test -p lpir
cargo test -p lpvm-cranelift
cargo test -p lpvm-native --features rt-jit
cargo test -p lpvm-native --features rt-emu
cargo test -p lpvm-wasm
cargo test -p lp-shader --features cranelift
```

## Dependencies

- M1 (pixel shader contract) — done
- LPIR inliner / stable function IDs from `feature/inline` — assumed
  landed before M2.0, not strictly required for M1.1 itself

## Why bundled

Each piece is small. Splitting into separate plans adds overhead
disproportionate to the work. They land together as "M2.0 prerequisites".
