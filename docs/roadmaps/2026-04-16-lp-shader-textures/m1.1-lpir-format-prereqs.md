# M1.1 — LPIR + Format Prerequisites

**Status: Done** (2026-04)

Implementation plan and notes: [`docs/plans/2026-04-17-lp-shader-textures-stage-iv/`](../../plans/2026-04-17-lp-shader-textures-stage-iv/).

## Goal

Land the small, self-contained changes that M2.0 (synthetic `__render_texture`)
depends on. Scope grew slightly during implementation: all narrow memory
operations landed (not only 16-bit store), plus both additional texture formats.

## What landed

### Narrow LPIR memory ops (six)

Word-aligned `Load` / `Store` were already present. Added:

- `Store8`, `Store16` — low 8 / 16 bits of an `i32` VReg to `[base + offset]`
- `Load8U`, `Load8S`, `Load16U`, `Load16S` — zero- or sign-extending narrow loads into `i32`

Backends: **Cranelift** (`lpvm-cranelift`), **native RV32** (`lpvm-native`), **WASM**
(`lpvm-wasm`), plus LPIR **interpreter**, **print/parse**, **validate**, and
`IrFunction::uses_memory`.

### `TextureStorageFormat`: `Rgb16Unorm` and `R16Unorm`

Alongside existing `Rgba16Unorm`:

- `Rgb16Unorm` — 3× u16 / 6 bpp  
- `R16Unorm` — 1× u16 / 2 bpp  

`bytes_per_pixel()` / `channel_count()` implemented for each variant (`lps-shared`).

### `compile_px` return-type validation

`expected_return_type` in `lp-shader` maps each format to the correct `LpsType`
(`Float` / `Vec3` / `Vec4`); tests cover acceptance and rejection per format.

### Stable function IDs (inliner groundwork)

Cherry-picked from the inline workstream: `b4a4e985`, `d096f8f6`, `1d6dedd1`.

### Dropped from milestone

**LPIR programmatic-construction spike** — not needed; `FunctionBuilder` and direct
`LpirOp` pushes are sufficient for upcoming synthesis work (see M2 roadmap).
