# Milestone 2.2: Legacy Project Sync Demolition

## Title And Goal

Delete the old project sync/detail/debug UI path after preserving it at tag `2026-05-07-pre-legacy-remove`.

## Suggested Plan Location

`docs/roadmaps/2026-05-06-slot-domain-cutover/m2.2-legacy-project-sync-demolition/`

## Scope

In scope:

- Remove the active `LegacyProjectResponse` / `LegacySerializableProjectResponse` project sync path.
- Remove legacy node detail requests, legacy node detail projection, and old `LegacyNodeState`-driven view plumbing from active code.
- Delete or quarantine node-specific debug UI panels that depend on legacy detail state.
- Keep resource summary/payload code if it is already useful for the canonical protocol.
- Keep source/engine/domain code that is still real, even if legacy sync previously consumed it.
- Let compile errors guide the rebuild boundary; do not preserve compatibility merely to keep old tests green.

Out of scope:

- Rebuilding canonical project sync messages.
- Rebuilding the project client/view.
- Rebuilding generic debug UI.
- Runtime state/params/output slot exposure.
- Client-driven mutation.

## Key Decisions

- The legacy implementation is preserved by tag/worktree, not by keeping old code alive in the active branch.
- Remaining legacy modules must be either deleted or clearly quarantined as reference-only/internal compatibility with a concrete reason.
- Resource payload selection remains a real concept and should be moved forward instead of casually deleted.

## Deliverables

- Active project sync no longer exposes or depends on legacy node detail types.
- Old debug UI detail panels are removed or disconnected from the build.
- The next milestone can define canonical project messages without fitting them through legacy protocol names.
- Notes record any legacy behavior that must be reintroduced intentionally in the canonical stack.

## Dependencies

- Tag `2026-05-07-pre-legacy-remove` exists as a reference point.
- Milestone 2 source defs expose real slot roots.
- Milestone 2.1 value leaves make inline value objects readable and serializable.

## Execution Strategy

Full plan. This milestone is intentionally destructive and will likely create compile fallout that is resolved in M3/M4. Keep commits small enough to make revert/review manageable.
