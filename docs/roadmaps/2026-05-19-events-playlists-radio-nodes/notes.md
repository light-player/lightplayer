# Events, Playlists, Button Nodes, And Radio Nodes Notes

## Scope Of The Effort

This roadmap covers the runtime/node layer that sits above the completed hardware IO substrate:

- Define how one-shot events work in the engine.
- Add an authored button input node that turns hardware button edges into graph events.
- Add a playlist/switching node that can select between multiple visual inputs.
- Add authored radio nodes that bridge local events/state to the shared radio API.
- Validate the first wireless button/sign behavior without turning this into a full event bus,
  show-control UX, or synchronized wireless state system.

## Current Codebase State

- Hardware capabilities exist and are root-owned:
  - `HardwareEndpointSpec` is the authored reference shape (`cap:driver:config`).
  - `ButtonDriver` / `ButtonInput` can emit `ButtonEvent`.
  - `RadioDriver` / `RadioDevice` support subscribe/send/drain by `RadioChannelId`.
  - `fw-esp32` registers ESP-NOW radio by default, and `fw-emu` has virtual radio support.
- The engine graph currently resolves frame-scoped values:
  - `LpValue` supports scalars, arrays, structs, resources, and lazy products.
  - `ProductRef` currently has `Visual` and `Control`.
  - Bindings connect literals, produced slots, consumed slots, and bus channels.
  - Resolver caching is per frame; produced slot values are sticky for the frame they are set.
- Runtime nodes tick through `NodeRuntime::tick(&mut TickContext)`.
  - `TickContext` can resolve consumed slots.
  - It does not currently expose hardware services directly.
  - `EngineServices` owns output/time services today and is the natural place to own button/radio
    provider handles or an event queue.
- `ButtonEvent` today is button-specific:
  - `ButtonEventKind::{Pressed, Released}`
  - `source: HardwareAddress`
  - `sequence: u32`
- `RadioMessageKind` today has `ButtonPress` and `Custom(u8)`, with a payload byte slice.

## User Notes And Direction

- The next work needs a new roadmap.
- Needed concepts:
  1. Event semantics, especially the event generated by a button.
  2. Playlist node for switching between multiple shader nodes.
  3. Button input node that forces the event model to become real.
  4. Radio node with shared event/state semantics and possibly bidirectional binding.
- The order above is conceptual, not necessarily the implementation order.
- Playlist could wait; a single shader node could accept events first if that proves simpler.
- The event should probably be more generic than `ButtonEvent`: a one-time event without payload
  could represent a button press, a motion sensor trigger, or another future edge.
- `TriggerEvent` is a good name for the first event family, but we expect other event families.
- OSC is appealing prior art. We should preserve a path toward OSC interoperability, but the first
  implementation should not include address or args because the immediate project is timeline-tight.
- The first fyeah sign behavior is not generic playlist "next". The button should bind to something
  like `bus#trigger`; pressing it twice should restart/retrigger the special effect, not advance
  back to the default state.
- Longer term, there will likely be standard UI bus channels/events such as `next`, `prev`, `pause`,
  `brightness`, and `speed`, but this project needs a trigger channel with restart semantics.
- The desired playlist shape is:
  - Item 1: default/idle look, loops forever under normal conditions.
  - Item 2: special animation, entered/restarted by `trigger`, plays for a configured duration, then
    returns to the normal repeating playlist/default state.
- The apparent naming overlap between `ControlProduct` and a possible `ControlMessage` is not
  necessarily a problem. DMX, MIDI, OSC, and LightPlayer control samples/messages are related
  domains. LightPlayer should grow toward being able to broker control data between protocols, such
  as accepting MIDI and producing DMX, or accepting OSC and driving internal playlist/control state.

## Initial Suggested Architecture

Use a unified control-domain lens with two related but separate terms:

- **Control message**: addressable typed message/event data, OSC-like and useful on the bus.
- **Control product/sample data**: durable or renderable control output data, such as fixture/DMX-ish
  sample buffers.
- **Event/trigger**: a one-shot control message with edge semantics. It is not a sticky boolean.
- **State**: durable control value that can be sampled, bound, serialized, and replayed.

The first implementation should bias toward an OSC-shaped event/message model, with
`TriggerEvent` as the first small event family:

```text
ButtonNode ── emits EventProduct / EventRef / EventValue
       │
       ├── PlaylistNode consumes events and selects VisualProduct input N
       │
       └── RadioSendNode serializes selected events to RadioMessage

RadioReceiveNode ── drains RadioDevice channel and emits events
```

The first event/message should carry only:

- `uid`: `u64`, unique enough for deduplication; by convention probably random.
- `source`: string, URI-like and semantic enough for display and debugging.
- `seq`: probably `u64`, source-local ordering/retrigger identity.

Important distinction: a button's raw hardware `Pressed/Released` edge can be translated into an
edge-shaped graph message while the lower-level hardware event can remain button-specific.

Deferred fields:

- `address`: omitted for now; implied by the bus channel/binding path in the first slice.
- `args`: omitted/commented out for now; will return when OSC-style typed arguments are needed.
- timestamp/timetag: omitted for now; source sequence plus frame observation is enough.

## Open Questions

### Q1. What is the first graph-level event type?

Suggested answer: introduce a generic `TriggerEvent` product/value rather than exposing
`ButtonEvent` directly. Keep `ButtonEvent` in hardware. Let `ButtonNode` map button edges to generic
graph events like `Pressed`, `Released`, or a configurable `Trigger`.

Reason: this keeps motion sensors, radio messages, and future trigger sources from inheriting
button vocabulary.

User answer: `TriggerEvent` is a good name. Other event families are expected later. OSC is useful
prior art, but for the near-term implementation omit address and args. Source string is enough.

### Q2. Should events be products, ordinary values, or OSC-shaped messages?

Suggested answer: define a minimal slotted `ControlMessage`/`TriggerEvent` envelope now, with the
future OSC-shaped fields intentionally deferred. Do not model a one-shot as `LpValue::Bool`.

Reason: the current resolver/binding model caches values per frame. A one-shot edge needs sequence
identity and "seen this frame" semantics. A sticky bool loses missed/repeated presses and makes
radio de-duplication awkward.

### Q3. Should button press/release both exist at graph level?

Suggested answer: support `Pressed` and `Released` as graph event kinds, but make the default button
node output a `Pressed` trigger because the first project needs a single action.

Reason: release events are useful for future hold/toggle behavior, but requiring consumers to handle
both immediately makes the first slice heavier.

### Q4. Should playlist come before radio nodes?

Suggested answer: event core -> button node -> minimal event consumer -> radio nodes -> playlist.

Reason: playlist is useful, but the harder and more central problem is how events move through the
runtime. A minimal test consumer can validate events before the visual switching node exists.

### Q5. Should radio be one node or split nodes?

Suggested answer: start with separate `RadioReceiveNode` and `RadioSendNode`, both using the same
endpoint/channel config. Later, consider a bidirectional `RadioNode` wrapper if authoring needs it.

Reason: send and receive have different directionality in the binding model. Splitting avoids
inventing bidirectional binding in the same milestone as first event semantics.

### Q6. How do events interact with shader code?

Suggested answer: do not feed raw event products into GLSL initially. Provide a small stateful
consumer node that converts events to shader-friendly state, such as a counter, selected index, or
momentary scalar.

Reason: GLSL wants current state, not an event stream. This keeps the JIT path simple and preserves
runtime shader compilation.

## Prior Art Notes

### MIDI

MIDI suggests a compact event vocabulary with:

- address-like routing dimensions (`group` / `channel` / endpoint in MIDI 2.0 terms),
- small typed event kinds (`NoteOn`, `NoteOff`, `ControlChange`, program changes, etc.),
- compact numeric identity for the controlled thing (`note`, `controller`, `program`),
- a small value field (`velocity`, controller value, pressure),
- optional timing messages/timestamps in richer transports.

Useful takeaways:

- Separate event identity from event payload.
- Keep simple trigger-like events cheap and fixed-size.
- Preserve explicit edge kinds: `on` and `off` are different messages, even if a common shortcut
  treats note-on-with-zero-velocity as off.
- Have a channel/topic dimension so independent controls do not collide.
- Do not make every input a boolean state; repeated presses need sequence/event identity.

Things not to copy too literally:

- The 7-bit data/value constraints.
- The hard-coded musical/channel vocabulary.
- The historical ambiguity around note-on-zero vs note-off.

### OSC

OSC suggests a path-addressed invocation model with:

- symbolic address paths such as `/oscillator/4/frequency`,
- typed argument lists,
- transport independence,
- bundles with timetags,
- dispatch by path/pattern matching.

Useful takeaways:

- A topic/path string is more authorable than a numeric controller ID.
- Event payloads can be optional and typed.
- Bundles/timetags are a future-friendly way to express "these happened together" or "apply at
  this time".
- Dispatch should not require the sender to know every receiver.

Things not to copy too literally:

- Full OSC pattern matching is likely too much for the first embedded graph.
- Variable argument blobs everywhere would make no_std validation and shader-state conversion
  harder than needed.

### Working Synthesis For LightPlayer

Use OSC as the long-term conceptual wire/authoring shape:

```text
ControlMessage / OscMessage {
  address,
  args,
}
```

But define the first slice as a tiny slotted envelope:

```text
TriggerEvent {
  uid: u64,
  source: String,
  seq: u64,
  // address: OscAddress,     // deferred; implied by bus channel for now
  // args: Vec<ControlArg>,   // deferred; OSC-style typed positional args later
}
```

Where:

- `uid` is globally unique enough for deduplication.
- `source` identifies the producing node/device/endpoint and is human-displayable.
- `seq` makes repeated triggers from the same source distinct and ordered.
- The bus channel supplies the first routing identity, e.g. `bus#trigger`.

Payload/address should be absent in the first slice. If we later need values, add OSC-compatible
typed positional args or a sibling payload-bearing message family rather than rushing it now.

## OSC Interop Implications

### Keep Addresses OSC-Compatible

Use OSC address syntax for event topics:

- Begins with `/`.
- Path segments are printable names.
- Avoid characters that OSC reserves for pattern matching or type tags in plain authored addresses
  unless we intentionally support patterns later.

This makes future UDP/TCP OSC input/output straightforward and keeps authoring familiar.

### Separate Message From Dispatch

OSC receivers dispatch a message by address. LightPlayer can mirror that internally:

- Nodes emit messages/events with addresses.
- Consumers subscribe/bind to exact addresses first.
- Pattern matching can be future work.

### Type Tags Later

First slice should support no-argument trigger messages only and omit address as an explicit field.
Later slices can add:

- bool/int/float/string arguments,
- typed values mapped to `LpValue`,
- bundles/timetags for scheduled or grouped events,
- OSC network import/export.

### Radio Mapping

Radio should not invent a separate semantic packet if we can avoid it. It can carry:

- channel ID at the radio layer,
- OSC-compatible address at the event/message layer,
- optional compact kind code for common trigger messages if packet size becomes a problem.

The first radio bridge can map `TriggerEvent.address` to/from radio payloads while keeping the
current shared `RadioMessage` framing for source device ID, event ID, and channel ID.

## Control Protocol Broker Direction

Longer term, LightPlayer should treat OSC, MIDI, DMX, radio messages, UI controls, and fixture
outputs as parts of a broader control-data domain:

```text
MIDI input ──► ControlMessage bus ──► DMX / fixture control output
OSC input  ──► ControlMessage bus ──► playlist / shader state
Button     ──► TriggerEvent        ──► radio / playlist / UI action
Radio      ──► ControlMessage bus ──► local behavior
```

This argues for names like `ControlMessage` even if `ControlProduct` already exists. The two names
can be made complementary:

- `ControlMessage`: discrete addressable message/event/control command.
- `ControlProduct`: renderable/sampleable control data product, currently used for fixture/output
  sample generation.

The roadmap should keep the first implementation small, but not choose names that block the broker
direction.

## Bus And Playlist Semantics

The existing bus can be the project-local event routing surface. For the first slice:

```text
ButtonNode ──produces── TriggerEvent { source, uid, seq }
      │
      └── binding target: bus#trigger

PlaylistNode trigger input ◄── binding source: bus#trigger
```

This makes the bus the named routing layer. For the first slice, the message does not carry an
address; `bus#trigger` is enough route/meaning. OSC-compatible address fields can be added when the
system needs external OSC interop or address-based dispatch.

Playlist behavior should distinguish at least:

- **Advance**: move to the next item from current state.
- **Trigger item**: enter/restart a specific item regardless of current state.
- **State control**: pause/resume/speed/brightness style values, likely not `TriggerEvent`.

For the fyeah sign, use **trigger item** semantics:

- idle item loops until interrupted;
- trigger starts special item from time zero;
- repeated trigger while special item is running restarts it from time zero;
- when special item's duration expires, playlist returns to idle/default item.

This avoids overloading `/next` and gives the later standard UI vocabulary room to mean ordinary
transport-like controls.
