# M2 Design: Static Shape Catalog And Dynamic Overlay

## Scope

Move static authored slot shapes out of per-engine heap residency. Static
shapes become generated borrowed descriptors that can live in flash. The runtime
registry becomes a dynamic overlay for project/runtime-generated shapes.

Out of scope:

- Host precompilation or removing on-device shader compilation.
- Keeping a firmware fallback that materializes every static shape into an
  owned `SlotShapeRegistry`.
- Rewriting generated slot views to fully precomputed accessors unless the main
  catalog migration is already stable.

## Architecture Summary

The static catalog is generated by `lpc-slot-codegen` from the model crate's
registered static slot shapes. Instead of calling
`StaticSlotShape::slot_shape()` and inserting owned trees into a registry, the
generated code exposes borrowed descriptor lookup by `SlotShapeId`.

`SlotShapeRegistry` keeps only dynamic entries and factories. Lookups resolve
dynamic entries first, then static catalog entries. This preserves dynamic
replacement behavior while removing resident heap for static built-ins.

Normal wire sync exports a conventional shape snapshot containing static
catalog shapes plus dynamic overlay shapes. Static shapes are serialized from
the generated catalog on demand for client setup; they are not inserted into
the engine registry. This keeps the device-side resident heap small without
adding a catalog fingerprint compatibility protocol.

## Main Components

### Static descriptors

`StaticSlotShapeDescriptor` mirrors `SlotShape` with borrowed data. The longer
name avoids colliding with the existing `StaticSlotShape` trait:

- field and variant names are `&'static str`
- metadata strings are `Option<&'static str>`
- collections are `&'static [...]`
- nested shapes are `&'static StaticSlotShape`

The descriptor family should be serializable through custom wire writers or
converted one shape at a time for tests/dev fallback. It should not require
allocating all static shapes at startup.

### Shape views

`SlotShapeView<'a>` represents either:

- `Static(&'static StaticSlotShapeDescriptor)`
- `Dynamic(&'a SlotShape)`

Traversal code should match on `SlotShapeView` helpers instead of assuming it
has an owned `SlotShape`.

### Lookup trait

`SlotShapeLookup` provides the common read surface:

```rust
fn revision(&self) -> Revision;
fn get_shape(&self, id: SlotShapeId) -> Option<SlotShapeView<'_>>;
fn contains_shape(&self, id: SlotShapeId) -> bool;
fn create_default(&self, id: SlotShapeId) -> Result<Box<dyn SlotMutAccess>, SlotFactoryError>;
```

`SlotShapeRegistry` implements this trait by checking dynamic entries first and
falling back to generated static catalog lookup.

### Static catalog

Generated API:

```rust
pub fn static_slot_shape(id: SlotShapeId) -> Option<&'static StaticSlotShapeDescriptor>;
pub fn static_slot_shape_name(id: SlotShapeId) -> Option<&'static str>;
pub fn static_slot_shape_ids() -> &'static [SlotShapeId];
pub fn create_static_slot_default(id: SlotShapeId) -> Option<SlotFactory>;
```

The old registration functions can remain temporarily for host tests, but the
engine path should stop using them before the milestone is complete.

### Wire behavior

`ShapeReadResult` and full slot sync should keep using the existing registry
snapshot payload shape. The exported snapshot includes static descriptors
generated one at a time from the static catalog plus any dynamic registry
entries. Internal registry snapshots can remain dynamic-only when the caller
does not need client setup data.

## Validation Strategy

Early phases validate with targeted model/codegen tests:

```bash
cargo test -p lpc-model slot_shape
cargo test -p lpc-slot-codegen
```

Later phases validate the server and emulator-facing path:

```bash
cargo check -p lpa-server
cargo test -p lpa-server --no-run
cargo test -p fw-tests --test profile_alloc_emu
```

Final validation should include M1 project-load profiles once available.
