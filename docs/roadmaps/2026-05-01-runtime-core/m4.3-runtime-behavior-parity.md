# Milestone 4.3: Runtime behavior parity

## Goal

Close the remaining MVP runtime behavior gaps between the old legacy runtime and
the core node path after sync and reload are working.

## Context

M4 exercises the simple shader -> fixture -> output path. It intentionally does
not yet model legacy multi-shader texture ownership/render order, deeper source
discovery, or all fixture behavior.

## In scope

- Support multiple shaders targeting one texture according to `render_order`.
- Decide the durable shared texture/render-product ownership model.
- Extend source discovery beyond immediate `/src` children if parity requires
  nested authored trees.
- Finish fixture mapping/color parity that is actually used by current projects.
- Remove or quarantine unused fixture transform config instead of porting it.
- Add parity tests that compare old-runtime reference behavior where useful.

## Out of scope

- Fixture transform config behavior. It was not used in practice and should not
  add runtime weight in the parity pass; treat future transform support as a
  separate visual feature.
- New visual node types beyond the MVP port.
- Legacy runtime removal; M5 owns that.

## Success criteria

- Existing authored MVP projects render the same visible output on the core path
  as on the old runtime for supported features.
- Multi-shader texture targets have deterministic behavior.
- Unused transform config no longer blocks or burdens the core runtime parity
  path.
