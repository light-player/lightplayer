# Phase 3: Streaming Base64 And Resource Payloads

## Scope Of Phase

Add streaming base64 support for binary JSON string fields and use it for runtime-buffer payload JSON.

In scope:

- Add a base64 writer that emits encoded bytes directly into the JSON stream.
- Add tests against known base64 strings and existing serde payload deserialization.
- Add a resource payload streaming helper for `WireRuntimeBufferPayload` or the resource result writer path.
- Ensure bytes are not converted into an intermediate encoded `String` or encoded `Vec<u8>`.

Out of scope:

- Chunked/ranged resource reads.
- Streaming every resource summary by hand if normal serde is cheap enough.
- ESP transport integration.

## Implementation Details

Base64 helper requirements:

- It writes the surrounding JSON string quotes or is clearly documented as writing only the encoded contents. Prefer an API that cannot accidentally omit quotes.
- It handles non-multiple-of-three byte lengths correctly.
- It writes padding correctly.
- It can operate on a borrowed `&[u8]` without allocating an encoded buffer.

Resource payload helper requirements:

- Preserve the JSON field names generated by serde:
  - `ref`
  - `revision`
  - `metadata`
  - `bytes`
- The streamed payload JSON should deserialize into `WireRuntimeBufferPayload`.
- For the first implementation, metadata can be serialized via serde bridge and bytes can use streaming base64.

Tests:

- `[]`, `[0]`, `[0, 1]`, `[0, 1, 2]`, and longer byte fixtures match `base64` crate output.
- Streamed `WireRuntimeBufferPayload` with bytes deserializes to the same struct as normal serde JSON.
- A resource result containing one runtime-buffer payload deserializes into `ProjectReadResponse` when written through the streamed response envelope.
