# M4: Control Flow — Notes

## Scope

Extend the fastalloc backward-walk allocator to handle **IfThenElse**, **Loop**,
and **Seq** regions. Currently the allocator only processes `Region::Linear`;
all other region types return `AllocError::UnsupportedControlFlow`.

After M4 the allocator handles any region tree produced by lowering. All
filetests pass under `rv32fa.q32`.

## Current State

### Region tree

Lowering already produces a full region tree (`lower_range` in `lower.rs`):

- `Region::Linear { start, end }` — instruction range `[start, end)`
- `Region::IfThenElse { head, then_body, else_body, else_label, merge_label }`
- `Region::Loop { header, body, header_label, exit_label }`
- `Region::Seq { children_start, child_count }` — children in `seq_children`

The lowerer:
- Emits `BrIf` in the head of an if (as a VInst)
- Emits `Br` (break/continue) in loop bodies
- Does NOT emit `Label` or `Br`-to-merge/back-edge — those are deferred to
  "the walker" (comments say walker will emit them)
- Stores `else_label`, `merge_label`, `header_label`, `exit_label` in the
  region nodes for the walker to use

### Allocator

- `fa_alloc::allocate()` only handles `Region::Linear` root. Any other root
  region → `UnsupportedControlFlow`.
- `walk_linear()` processes straight-line code with backward walk.
- Liveness analysis (`analyze_liveness`) already recurses over all region types
  but `live_out` is always empty. `live_in` is a conservative union.

### Test status (rv32fa.q32)

- **3631 pass**, 1512 fail, 196 compile-fail, 0 unsupported
- 436/691 files pass
- **196 compile-fail files** — these are the control flow failures
  (`UnsupportedControlFlow`) plus some unimplemented lowering ops
- **59 runtime failure files** — separate bugs (matrix ops, bit ops, etc.)
  not related to control flow

### What the old backend does (lpvm-native)

- Flat VInst stream with explicit `Label`, `Br`, `BrIf` VInsts
- Linear scan allocator on the flat stream
- `extend_for_loops` widens live ranges for loop-carried values
- No boundary spills — relies on intervals spanning across branches

### Key architectural difference (FA approach)

The FA allocator uses region-tree recursive walks with **spill-at-boundary**:
- At each control flow boundary, all live-in-register values are spilled
- Re-entering a region reloads from spill slots
- This avoids parallel move resolution at merge points
- Cost: more memory traffic, but acceptable for shaders

## Questions

### Q1: Should the walker emit Label/Br VInsts, or emit them as machine code directly?

**Context**: The lowerer comments say "Labels and J are emitted by walker, not as
VInsts." This means the walker (allocator) needs to emit branch/label machine
code. But the current architecture has the emitter (rv32/emit.rs) handle VInsts.

**Answer**: The **emitter** handles labels and branches at region boundaries based
on the region tree. The allocator only handles register state and edits. This
matches regalloc2's approach — control flow instructions are part of the
instruction stream, not generated by the allocator. The emitter will walk the
region tree and emit Label/Br/BrIf machine code at the appropriate points
around allocated region bodies.

### Q2: Boundary cost and performance

**Context**: Spill-at-boundary means every value live across a control flow
boundary goes through a store/load pair. Is this worse than regalloc2?

**Answer**: No — regalloc2 fastalloc has the same cost. Its invariant is
"liveins arrive in spillslots" at every block boundary. Both approaches pay
one sw + one lw per live value per boundary crossing.

Two cheap optimizations to implement in M4:

1. **On-demand reload**: Start each region with an empty pool. The backward walk
   only reloads values actually used in that region. Free.
2. **Liveness-guided boundary spills**: At a boundary, only spill values in
   `live_in` of the destination region. Values that die at the boundary just get
   freed (no store). Uses existing `analyze_liveness`.

Future optimization (M5+): keep loop-invariant values in callee-saved registers
across iterations.

### Q3: How should boundary edits be represented?

**Context**: At each boundary (if entry/exit, loop entry/exit), we need to
spill live registers. The current `Edit` enum has `Move { from, to }` and
`LoadIncomingArg`.

**Answer**: Existing `Edit::Move` with `EditPoint::After(last_inst_in_region)`
is sufficient for boundary spills (reg → slot). Reloads are implicit — the
backward walk's `alloc_use` inserts them naturally when a spilled value is used.
No new Edit variants needed.

### Q4: Loop continue semantics

**Context**: The lowerer emits `Br(continuing)` for `Continue` statements, but
`Label(continuing)` is deferred to "the walker." This creates an unresolved
branch target.

**Answer**: The lowerer should emit `Label(continuing)` as a VInst at the right
position in the body. The allocator treats Label as a no-op. The emitter records
it for branch resolution. This is what the old backend does and keeps
`Br(continuing)` / `Label(continuing)` self-consistent in the VInst stream.

### Q5: Pool state threading between regions

**Answer**: At boundaries, spill everything live (using liveness to skip dead
values). Each region starts with an empty pool and reloads on demand during the
backward walk. This matches regalloc2's boundary invariant and avoids needing
to thread or merge pool states between regions. For Seq, walk children in
reverse order; the pool is empty between children.

### Q6: How should `allocate()` dispatch to the region walker?

**Answer**: Add a `walk_region(tree, region_id, ...)` recursive function that
dispatches on region type:
- Linear → existing backward walk logic
- Seq → walk children in reverse order, boundary spill between each
- IfThenElse → spill at merge, walk else (empty pool), spill, walk then
  (empty pool), spill, walk head
- Loop → spill at exit, walk body (empty pool), spill at back-edge, walk
  header (empty pool), spill at entry

`allocate()` calls `walk_region(root)`.

### Q7: Emitter region tree walk

**Answer**: The emitter walks the region tree in forward order, emitting
labels/branches at boundaries from the Region node's stored LabelIds:
- IfThenElse: head vinsts → then body → Br(merge) → Label(else) → else body
  → Label(merge)
- Loop: Br(header) → Label(header) → header → body → Br(header) → Label(exit)
- Seq: emit children in order
- Linear: emit vinsts in range

Between regions, the emitter processes boundary edits from the alloc output.

### Q8: Alloc output format

**Answer**: Keep the flat `allocs` array unchanged. VInst indices are global
across all regions. `inst_alloc_offsets[i]` works as-is — the walker populates
it region by region during the recursive walk.

## Notes

- regalloc2 fastalloc has the same boundary cost as our spill-at-boundary
  approach. Both require liveins to be in spillslots at block/region boundaries.
- Loop-invariant register pinning is a future optimization (M5+).
- The 196 compile-fail files are mostly `UnsupportedControlFlow` plus some
  unimplemented lowering ops. The 59 runtime failures are separate bugs.
