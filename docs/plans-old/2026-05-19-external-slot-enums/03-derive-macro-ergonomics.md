# Phase 3: Derive Macro Ergonomics

## Scope of Phase

Add derive macro support for opting enum slots into external encoding and for Serde-like variant renaming.

In scope:

- Add `#[slot(enum_encoding = "external")]` on enum containers.
- Add `#[slot(rename_all = "snake_case")]` on enum containers.
- Keep `#[slot(name = "...")]` as an explicit per-variant override.
- Add derive tests that prove generated shapes and authored TOML behavior.

Out of scope:

- Supporting every Serde rename policy.
- Field-level `#[slot(key)]`.
- Applying `rename_all` to record fields.

## Implementation Details

Relevant files:

- `lp-core/lpc-slot-macros/src/attr.rs`
- `lp-core/lpc-slot-macros/src/slotted_enum.rs`
- `lp-core/lpc-model/tests/slotted_enum_derive.rs`

Expected changes:

1. Extend enum/container attribute parsing:

   ```rust
   #[slot(enum_encoding = "external")]
   #[slot(rename_all = "snake_case")]
   ```

2. Accept only supported values:

   - `enum_encoding = "tagged"` or omitted means default tagged encoding.
   - `enum_encoding = "external"` produces external encoding.
   - `rename_all = "snake_case"` is supported.

3. Reject unsupported values with clear compile errors.

4. Update `slotted_enum.rs` so generated `SlotShape::Enum` includes the selected encoding.

5. Apply variant names in this precedence order:

   - Explicit `#[slot(name = "...")]`.
   - `rename_all = "snake_case"` transformed Rust variant name.
   - Existing Rust variant name string.

6. Add a small helper in the macro crate for `PascalCase` / `CamelCase` to `snake_case`.

   Keep it ASCII-focused to match slot name validation.

Tests to add:

- Existing enum derive still produces tagged `kind` encoding by default.
- External enum derive produces `SlotEnumEncoding::External`.
- `rename_all = "snake_case"` maps `OptionA` to `option_a`.
- Explicit `#[slot(name = "custom")]` overrides rename-all.
- External enum generated by derive reads/writes TOML through the dynamic slot codec.

Edge cases:

- Do not change field names for record payload fields in this phase.
- Do not change existing `NodeDef` variant names or syntax.
