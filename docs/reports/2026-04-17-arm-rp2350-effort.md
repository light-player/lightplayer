# ARM RP2350 Backend — Effort Analysis

**Date:** 2026-04-17
**Context:** Adding a Cortex-M33 / RP2350 target to the `lpvm-native` compiler backend, alongside the current RV32IMAC / ESP32-C6 path.

---

## TL;DR

Doable in roughly the same shape as the original `lpvm-native` build — realistic estimate **5–8 calendar weeks** at the team's current pace. The biggest single line-item is a new emulator, not a new emitter. The architecture is already most of the way to multi-ISA; the work is "rewrite the leaves, refactor the trunk."

Targeting RP2350 (Cortex-M33, Thumb-2, 520 KB SRAM, 4 MB flash, dual-core 150 MHz) sidesteps the ARMv6-M / Cortex-M0+ pain entirely. M33 with Thumb-2 is much closer to RV32IMAC in emitter difficulty.

---

## What's actually portable today

The code is more RISC-V-shaped than the doc comments admit, but the boundaries are clean enough.

### Fully reusable (no work)

- `lpir` + `lps-frontend` (GLSL → LPIR via naga)
- `lower.rs` (1.9k LOC, only 2 trivial `rv32` references). `VInst` is documented as "RISC-V shaped" but in practice the ops (`AluRRR`, `AluRRI`, `Load32/Store32`, `Mov`, `Br/BrIf`, `Call`, `Ret`, `Label`, `Icmp`, `Select`, `MemcpyWords`) are vanilla RISC ops that map fine to Thumb-2.
- `region.rs`, `regset.rs`, `vinst.rs`
- Q32 fixed-point float strategy — no FPU needed, soft-float ABI all the way.

### Mostly reusable (1–3 days of refactor)

- `regalloc/{walk,liveness,spill,pool,verify}.rs` (~3.4k LOC). The walk-allocator is target-agnostic in its bones, but it imports `crate::rv32::gpr::PReg` (a `u8`) directly, and `Alloc::Reg(PReg)` uses that type. There are only ~3 hard-coded touches in `walk.rs` (caller-saved-pool check, `SRET_SCALAR_THRESHOLD`).

  The good news: `crate::abi::PReg` already exists as a `(class, hw)` struct, designed for multi-ISA use. The fix is mechanical — replace `rv32::gpr::PReg` with `abi::PReg` everywhere in `regalloc/`.

- `abi/frame.rs`, `abi/classify.rs` — generic, just need to be parameterized on stack alignment (RV32 = 16, AAPCS = 8).
- `abi/func_abi.rs` — `ModuleAbi::from_ir_and_sig` hard-codes `func_abi_rv32`; needs an arch dispatch (trait or enum).

### Must be written from scratch

- `rv32/encode.rs` (374 LOC) → `arm/encode.rs`. Thumb-2 encoder is comparable size.
- `rv32/emit.rs` (976 LOC) → `arm/emit.rs`. **This is the bulk of the work.** The structure is reusable (prologue/epilogue, edit emission, label fixup, branch fixup) but every instruction sequence is rewritten. `emit_icmp` for instance maps to a different idiom (cmp + IT + cset on M33).
- `rv32/abi.rs` (461 LOC) → `arm/abi.rs`. AAPCS: r0–r3 args / r0–r1 return / r4–r11 callee-saved / r12=ip / r13=sp / r14=lr / r15=pc. Sret in r0, like RV. Probably ~60% smaller than the RV32 version because AAPCS has fewer arg regs and a simpler classification.
- `link.rs` patch logic — RV32 uses `auipc+jalr` pairs with `R_RISCV_CALL_PLT`. ARM uses `BL` with `R_ARM_THM_CALL`. ±16 MB range on Thumb-2 BL, so probably never need long-call thunks for shader-sized images, but the patcher still needs to know about the encoded immediate split (Thumb BL is a 32-bit instruction with the immediate split across two 16-bit halves).

### The big one: a new emulator

`lp-riscv-emu` is ~14k LOC. About 4k of that is generic infra (memory model, logging, error handling, function-call glue, ABI helpers, backtrace, debug) that should be factored into a shared crate now and reused.

The ISA executors (atomic 861 + compressed 1.6k + immediate 1.2k + arithmetic 1.5k + branch 335 + jump 257 + load_store 635 + system 449 ≈ 7k LOC) all need ARM equivalents. **But** you only need the subset your emitter actually produces — same trick `lp-riscv-emu` already uses. Realistic Thumb-2 subset: **2.5–4k LOC**.

## ARM-specific risks

### 1. Builtins and Cranelift

Builtins (psrdnoise, q32 math, etc.) are pre-compiled via Cranelift in `lpvm_cranelift::link_object_with_builtins` and linked into the JIT image. **Cranelift has no 32-bit ARM backend** (only AArch64).

- **On device:** Doesn't matter. Compile the builtin Rust crate with `rustc --target=thumbv8m.main-none-eabi` and ship the object as part of the firmware, exactly like RV32 does.
- **For host emulation (`rt_emu`):** You currently lean on Cranelift to AOT the builtins into native code that the emulator runs. You'd need to either build the thumbv8m object on host with `cc`/`rustc` and load it into the ARM emulator (clean), or stretch your own ARM emitter to handle the builtins (much more work). The first option is the right call — maybe 1–2 days of build-glue.

### 2. VInst's RISC bias is mild but real

Two examples:

- `MemcpyWords` is fine on ARM (LDM/STM is great), but lowering currently emits the RISC-V idiom — you may want a small Thumb-2-specific lowering pass for it.
- `IConst32` on RV32 lowers to `lui+addi`. On Thumb-2 you have `MOVW+MOVT` (clean and constant-time). No literal pool needed for 32-bit constants. This is actually *easier* than RV32, not harder.

### 3. Branch encoding differences

Thumb-2 conditional branches: ±1 MB. Unconditional B: ±16 MB. BL: ±16 MB. All comfortable for shader-sized images, so no long-branch islands needed. The branch-fixup path in `rv32/emit.rs` translates directly.

### 4. IT blocks

ARM Thumb-2's `IT` (if-then) blocks let you predicate up to 4 instructions. The `Select` and `Icmp` lowerings will benefit from these — short conditional sequences without explicit branches. Worth modeling, but not strictly required for a first cut (you can always emit `cmp + b<cond>` like RISC-V does).

## Realistic effort estimate

Calibrating against "compiler backend in <2 weeks":

| Task | Days |
|---|---|
| Generalize regalloc to `abi::PReg`, arch-dispatch `ModuleAbi` | 2–3 |
| Thumb-2 encoder + emitter (mirror of `rv32/`) | 5–8 |
| AAPCS ABI + frame tweaks | 2 |
| BL relocations, link.rs split | 2–3 |
| ARM subset emulator (after factoring shared infra out of `lp-riscv-emu`) | 6–10 |
| Builtins: thumbv8m `cc`-built object loading for `rt_emu`; firmware-side relink for `rt_jit` | 2–3 |
| `rt_emu` mirror, filetests, port `rv32n.q32` filetest target → `armn.q32` | 3–4 |
| Slop / debugging / tests | 4–7 |
| **Total** | **~26–40 working days = 5–8 calendar weeks** |

## Recommended preparatory refactors (do these first regardless)

These are pure architectural wins that pay off before any ARM code is written and probably take ≤1 week total.

1. **Promote `crate::abi::PReg` to be the canonical type in regalloc.** Delete the `rv32::gpr::PReg = u8` alias from the regalloc layer. This is a 1-day refactor that immediately makes the allocator ISA-agnostic and removes the `crate::rv32::abi::SRET_SCALAR_THRESHOLD` reference in `walk.rs`.

2. **Extract generic infra from `lp-riscv-emu` into a `lp-emu-core` crate.** `Memory`, `EmuSharedArena`, `LogLevel`, `format_debug_info`, `call_function_with_struct_return`, `abi_helper.rs`. About 3–4 k LOC of code that's not RISC-V-specific. This makes a future `lp-arm-emu` a much smaller crate and keeps `lp-riscv-emu` honest.

3. **Split `link.rs` into a generic `relocate` driver and an arch-specific `patch_*` callback.** Currently `patch_call_plt` is hardwired into `link_jit`.

After these three, the actual ARM port is a known-shape problem.

---

## Why RP2350 is the right pick

- 520 KB SRAM and 4 MB flash — strictly more headroom than ESP32-C6 (512 KB / 4 MB).
- 150 MHz dual-core M33 — comparable single-thread perf to ESP32-C6 at 160 MHz; second core is a bonus for parallel render or async LED output.
- Thumb-2 has hardware `sdiv`/`udiv`, wide ALU encodings addressing all of r0–r12, conditional execution via `IT`, and BL with ±16 MB range. All of this maps naturally to the IR shapes we already produce.
- Soft-float Q32 path stays unchanged.
- Toolchain (`thumbv8m.main-none-eabihf` / `thumbv8m.main-none-eabi`) is a first-class Rust target.

---

## Bottom line

Not a small project, but the same shape as `lpvm-native` was. The biggest single line-item is the new emulator, not the new emitter. The three preparatory refactors are valuable on their own and convert this from "ARM port" into "drop in a new emitter + ABI + ISA executor against an already-multi-ISA core."
