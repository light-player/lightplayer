# RV32 Load16 Issue Report

**Date:** 2026-04-27
**Target:** ESP32-C6, `rv32imac-unknown-none-elf`
**Context:** M3b `texelFetch` texture reads introduced LPIR `Load16U` on runtime texture data. The `lpvm-cranelift` RV32 path currently lowers `Load16U` through an explicit word-load decomposition instead of Cranelift's ordinary `uload16` path.

## Executive Summary

The current `lpvm-cranelift` workaround is a code smell because it makes `Load16U` semantics depend on the backend. It implements byte-addressable, potentially unaligned 16-bit reads by lowering to aligned 32-bit loads and bit extraction. That is stronger than what the current texture path needs, and it is not what `lpvm-native` emits.

For the current `texelFetch` formats, a normal RV32 `lhu` is sufficient if the texture descriptor points to the start of a texture allocation:

- Filetest texture allocations use 4-byte alignment.
- `R16Unorm`, `Rgb16Unorm`, and `Rgba16Unorm` all have even bytes-per-pixel.
- Channel offsets are `channel_index * 2`.
- Therefore every generated texture channel address is 2-byte aligned.

The bigger issue is test fidelity. The emulator enforces aligned halfword and word accesses by default, which is good. However, it does not appear to gate decoded instructions by an explicit `rv32imac` feature set. It supports several Zba/Zbb/Zbs bitmanip instructions even though those are outside `rv32imac`. That can produce false successes if either backend emits instructions the ESP32-C6 target cannot execute.

## RV32IMAC Load Constraints

RV32I includes the scalar load instructions relevant here:

- `lb` / `lbu`: byte loads, sign- or zero-extended.
- `lh` / `lhu`: 16-bit halfword loads, sign- or zero-extended.
- `lw`: 32-bit word load.

The `M`, `A`, and `C` extensions do not change the existence of `lh` or `lhu`; they are base RV32I instructions. A backend does not need Zcb or any other code-size extension to load 16-bit texture channels.

Natural alignment is the portable rule:

- `lh` / `lhu` addresses should be divisible by 2.
- `lw` addresses should be divisible by 4.
- Byte loads have no alignment issue.

The RISC-V unprivileged ISA leaves misaligned load/store handling to the execution environment. Some environments complete them, some trap, and some are slow or partially emulated. For LightPlayer's on-device compiler output, generated code should not rely on misaligned halfword or word loads unless we have explicitly decided that ESP32-C6 supports the exact behavior we need and our emulator models it.

Compressed halfword load forms such as `c.lhu` and `c.lh` are not part of plain RV32IMAC. They belong to Zcb, a separate code-size reduction extension layered on top of Zca. The ordinary 32-bit `lhu` instruction is legal on RV32IMAC; the compressed Zcb `c.lhu` form is not.

## LPIR alignment contract (chosen)

**Chosen contract: natural alignment.** The source of truth is the doc comments on each load in `lp-shader/lpir/src/lpir_op.rs`:

- `Load8U` / `Load8S`: no alignment requirement (any byte address).
- `Load16U` / `Load16S`: `base + offset` must be 2-byte aligned.
- `Load` / `Store` of 32-bit words: `base + offset` must be 4-byte aligned.

Backends and tooling may still be **permissive internally** (e.g. `lpir::interp` can read two bytes at any offset with `u16::from_le_bytes`; `lpvm-wasm` can emit WASM `i32.load16_u`, which allows unaligned accesses in the WASM memory model). **Generated LPIR is still expected to satisfy the contract above** so that strict targets (RV32 `lh`/`lhu`/`lw`, Cranelift lowerings, emulator strict mode) remain valid.

Texture fetches and current filetests already use layouts that satisfy halfword alignment for channel reads; the open work is tightening backends and tests around this contract, not re-litigating whether odd-address `Load16U` is part of the IR.

**Avoid introducing byte-addressable `Load16` behavior as the default** unless a future use case needs a distinct operation (e.g. an explicit unaligned narrow load); the recommended path is to keep narrow halfword loads naturally aligned in LPIR and fix lower layers if something fails.

## Emulator Findings

`lp-riscv-emu` models load alignment strictly by default:

- `Memory::read_halfword` rejects odd addresses unless `allow_unaligned_access` is set.
- `Memory::read_word` rejects non-4-byte-aligned addresses unless `allow_unaligned_access` is set.
- The `Riscv32Emulator` constructor leaves `allow_unaligned_access` disabled by default.

That is the right default for catching backend bugs intended for RV32IMAC portability.

There are call sites elsewhere that deliberately enable unaligned access for ESP32-style behavior, but the LPVM filetest paths inspected here create emulators with the strict default. That means the texture filetests should catch accidental odd `lhu` addresses on `rv32n.q32` or `rv32c.q32`.

The fidelity risk is instruction-set gating, not load alignment. The emulator dispatches by opcode and implements at least some instructions beyond RV32IMAC:

- Opcode `0x2f` executes A-extension atomics, which are valid for `rv32imac`.
- The immediate and R-type paths include Zba/Zbb/Zbs bitmanip instructions.
- Compressed decoding is enabled whenever instruction bits indicate a 16-bit instruction, without an explicit target profile check.

The compressed decoder currently covers standard RVC forms such as `c.lw`, `c.sw`, `c.lwsp`, and arithmetic/control-flow forms. It does not appear to implement Zcb `c.lhu`, `c.lh`, or `c.lbu` today. That means a backend emitting Zcb narrow compressed loads should fail in the emulator, not falsely pass. The broader false-success risk remains for bitmanip and any future non-profile instructions added to the emulator without feature gating.

## `lpvm-native` Assessment

`lpvm-native` lowers narrow loads directly to ordinary RV32 instructions:

- `Load8U` -> `lbu`
- `Load8S` -> `lb`
- `Load16U` -> `lhu`
- `Load16S` -> `lh`
- `Load32` -> `lw`

For the current texture path, this is correct with the alignment facts above. A texture base allocated at 4-byte alignment plus even row, pixel, and channel offsets gives a 2-byte-aligned `lhu` address.

The backend is not a byte-addressable unaligned `Load16U` implementation. If LPIR later requires odd-address `Load16U` to succeed on RV32, `lpvm-native` would need either a decomposition sequence or a stronger IR validation rule that rejects/avoids misaligned uses.

## `lpvm-cranelift` Assessment

Cranelift's RV32 lowering has an ordinary `uload16` rule:

```text
uload16 -> LoadOP.Lhu
sload16 -> LoadOP.Lh
```

The RISC-V emitter also has a compression path for `LoadOP::Lhu`, but only when `has_zcb` is enabled. Since Zcb is distinct from RV32IMAC, this gating is conceptually correct. The important thing to verify is that our target configuration never enables `has_zcb` for ESP32-C6 or filetest RV32 targets.

The current `lpvm-cranelift` object path has a target mismatch worth cleaning up:

- `object_module.rs` constructs `Riscv32imafc`, even though the product target is `rv32imac`.
- The Cranelift RV32 settings default `has_m` and `has_a` to true, but `has_zca` and `has_zcb` to false.
- The object path does not explicitly configure a precise ESP32-C6 feature set.

Because `has_zcb` defaults false, the specific `c.lhu` false-success risk is probably not active today. But the target setup is still too implicit for a safety-sensitive backend test: it should say `rv32imac` and explicitly enable the target extensions we intend to test, including C/Zca, while keeping Zcb disabled.

The `Load16U` workaround in `emit/memory.rs` is broader than needed and changes semantics:

- It uses aligned 32-bit `load` operations and bit extraction.
- It allows odd halfword addresses.
- It turns one legal `lhu` into a much larger sequence.
- It bypasses the actual Cranelift `uload16 -> lhu` path we need confidence in.

This makes it a poor long-term fix. If plain `uload16` fails, the failure should be reproduced and fixed in `lp-cranelift` or in the emulator, not hidden in the LPIR-to-CLIF adapter.

## Current Texture Path

For M3b `texelFetch`, the generated address is:

```text
addr = descriptor.ptr + y * row_stride + x * bytes_per_pixel + channel * 2
```

The supported texture formats are:

- `R16Unorm`: 2 bytes per pixel.
- `Rgb16Unorm`: 6 bytes per pixel.
- `Rgba16Unorm`: 8 bytes per pixel.

Filetest fixtures allocate texture storage with 4-byte alignment. Because every additive term after the base is even, texture channel addresses are halfword aligned. Therefore RV32 `lhu` is the right target instruction for texture channel reads.

## Risks

### False Success: Non-IMAC Instructions

The emulator currently supports at least some non-IMAC instructions. If `lpvm-native`, `lpvm-cranelift`, or the linked builtins emit Zba, Zbb, Zbs, or other non-target instructions, emulator tests may pass while ESP32-C6 execution fails.

This is higher priority than the `Load16U` workaround because it affects all RV32 backend validation, not just textures.

### False Confidence: Workaround Avoids `lhu`

The `lpvm-cranelift` workaround means `rv32c.q32` texture tests no longer prove that Cranelift can emit and run a legal `lhu` path. They prove only that our custom decomposition works for those data layouts.

### Contract Drift: LPIR Alignment

The contract is now documented on `LpirOp` (`Load16*` requires 2-byte alignment; word `Load`/`Store` requires 4-byte alignment). Residual risk is implementation drift: future changes could still generate misaligned narrow loads while relying on permissive interpreter or WASM paths, then fail on device—validation and targeted tests remain important.

## Recommendations

1. LPIR narrow-load alignment semantics are documented in `lpir_op.rs` as natural alignment (`Load8*` unrestricted; `Load16U` / `Load16S` require 2-byte alignment; word `Load`/`Store` require 4-byte alignment). Follow-on work is validation and targeted tests where practical, not redefining the contract.

2. Remove the `lpvm-cranelift` `Load16U` decomposition unless a focused failing test proves it is required.

   The texture path should use ordinary `uload16 -> lhu`. If that fails, capture the failure as a minimal Cranelift/LPVM test and fix the lower layer.

3. Add emulator ISA-profile gating for `rv32imac`.

   The emulator should reject instructions outside the configured target profile. At minimum, default filetest execution should allow IMAC and reject bitmanip instructions unless the test explicitly selects a target that includes them.

4. Make Cranelift RV32 target configuration explicit.

   The object path should use `rv32imac`, not `riscv32imafc`, for ESP32-C6-equivalent validation. It should explicitly set or assert relevant Cranelift flags: M/A/C/Zca on, Zcb off, and F/D off.

5. Add focused guardrail tests.

   Add small tests that:

   - Execute `lhu` at an even address and pass.
   - Execute `lhu` at an odd address and fail in strict `rv32imac` emulation.
   - Feed a known non-IMAC instruction to the `rv32imac` emulator and require rejection.
   - Confirm `lpvm-native` and `lpvm-cranelift` texture channel reads use aligned addresses.

## Bottom Line

The current texture feature does not require an unaligned `Load16U` workaround. Normal RV32 `lhu` is legal and appropriate for aligned `u16` texture channels on RV32IMAC.

The thing to fix before trusting device behavior is the validation boundary: the emulator and Cranelift target setup should explicitly model `rv32imac`, so backend tests cannot pass with non-target instructions or accidental unaligned memory semantics.
