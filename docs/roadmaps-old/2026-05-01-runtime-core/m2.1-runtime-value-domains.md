# Milestone 2.1: Runtime value domains

## Goal

Introduce the core runtime value/domain shape that lets engine resolution return
more than scalar `LpsValueF32` values, without implementing the full render
product system yet.

This milestone bridges M2's engine-owned `ProducedValue` work and M3/M4's legacy
source/runtime migration. The immediate pressure is that visual outputs already
want to behave differently from simple scalar/vector values: textures have been
forced through `LpsValue`, and future sampled render products should not repeat
that mistake.

## Context

M2 creates the first core `Engine`, `Resolver`, `ResolveSession`, `QueryKey`, and
`ProducedValue` path. The first implementation can resolve ordinary values, but
the runtime direction needs an explicit domain boundary before legacy shader /
fixture / output behavior is ported onto the new engine.

The long-term model should resemble a small demand-driven dataflow engine with
domain-specific produced values:

- simple control/model values;
- texture-backed visual products;
- sampled or batch-sampled visual products;
- later audio/channel/geometry/table-like products if the desktop/GPU runtime
  grows in that direction.

M2.1 should define the first engine-owned envelope and type vocabulary, not the
entire future product family.

## In scope

- Rename or reshape `ProducedValue` so the resolver cache stores a versioned
  runtime value envelope instead of only `Versioned<LpsValueF32>`.
- Add a small engine-owned runtime value enum, initially with a scalar/simple
  value variant and a placeholder/render-product-ready variant only if needed.
- Decide the crate boundary between portable value/domain descriptors in
  `lpc-model` and engine-only runtime handles in `lpc-engine`.
- Preserve M2 behavior for ordinary value resolution.
- Make the new shape ready for M3/M4 legacy migration to represent texture or
  render-product outputs without embedding those products in `LpsValue`.
- Add focused tests for scalar compatibility, value domain/type reporting, and
  resolver cache behavior with the new envelope.

## Out of scope

- Implementing sampled render products.
- Implementing a full render product trait/object registry.
- Porting legacy shader, fixture, texture, or output runtimes.
- Adding GPU-backed desktop product storage.
- Adding async or parallel resolution.
- Adding a full type checker for the node graph.

## Key decisions to make during planning

- **Value envelope name:** whether the runtime enum is called `RuntimeValue`,
  `ProducedData`, `EngineValue`, or another name.
- **Crate ownership:** whether domain/type descriptors live in `lpc-model` while
  runtime handles live in `lpc-engine`.
- **Render placeholder:** whether M2.1 should add only the scalar variant, or
  also add a non-functional render-product handle variant to prevent scalar-only
  assumptions from landing in M3/M4.
- **Compatibility path:** how much existing `LpsValueF32` API surface should
  remain as helper methods versus being updated immediately.

## Success criteria

- `ProducedValue` no longer hard-codes `Versioned<LpsValueF32>` as the only
  produced data shape.
- Ordinary scalar/simple value resolution still works through the M2 engine.
- The new runtime value/domain shape makes it clear where future texture-backed
  and sampled render products will fit.
- M3 and M4 can proceed without putting texture/render-product concepts inside
  `LpsValue`.
