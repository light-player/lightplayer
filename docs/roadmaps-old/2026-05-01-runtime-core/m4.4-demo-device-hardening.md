# Milestone 4.4: Demo and device hardening

## Goal

Validate the parity-complete core runtime on desktop demo and embedded/device
flows before removing the legacy runtime.

## Context

M4 made the core runtime the active server/demo path, but manual demo and device
validation should happen after sync, reload, and runtime behavior parity are in
place. M4.4 is the stabilization gate before M5.

## In scope

- Run desktop `just demo` and fix integration fallout.
- Run the required ESP32 firmware check with compiler included.
- Run focused emulator/device validations that exercise shader compilation,
  fixture mapping, and output flushing.
- Audit memory/copy behavior introduced by render products and runtime buffers.
- Update docs with the final pre-M5 validation results and any accepted risks.

## Out of scope

- Broad deletion of old runtime code; M5 owns that.
- New sync contracts or runtime features beyond hardening fixes.
- Cosmetic client polish unrelated to parity validation.

## Success criteria

- Desktop demo shows loaded nodes with state/details and visible output data.
- ESP32 check passes with the embedded JIT path intact.
- No known M4.x parity blocker remains that would make legacy removal risky.
