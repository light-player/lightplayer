# Milestone 4.2: Source reload and lifecycle parity

## Goal

Make the core runtime respond to authored source changes and lifecycle events
with enough parity that M5 can remove the legacy runtime without hiding reload
or teardown regressions.

## Context

M4 cut the active server/demo path over to `CoreProjectRuntime`, but
`handle_fs_changes` is currently a no-op and unload/stop-all drops the runtime
without explicit node destroy or output close traversal.

## In scope

- Rebuild affected core nodes when `node.toml` changes.
- Recompile shader nodes when GLSL source changes.
- Remove or mark nodes dead when authored node files/directories are deleted.
- Add explicit destroy traversal for core nodes.
- Unregister output sinks and close provider handles during teardown.
- Preserve filesystem version tracking behavior in `lpa-server`.
- Add scene update tests that assert reload, recompile, deletion, and teardown.

## Out of scope

- Runtime buffer/render-product detail sync; M4.1 owns that.
- Multi-shader/shared texture render order; M4.3 owns that.
- Removing `LegacyProjectRuntime`; M5 owns that.

## Success criteria

- Existing scene update tests can assert real source changes instead of the M4
  no-op behavior.
- `UnloadProject` and `StopAllProjects` release output resources explicitly.
- Server filesystem version tracking still avoids repeated changes.
- The core runtime can survive common edit/reload loops without restarting the
  whole server.
