# Milestone 4.1: Runtime buffer and detail sync projection

## Goal

Restore client-visible detail/state parity on top of the M4 core runtime while
replacing legacy heavy-state buffer snapshots with buffer-aware sync semantics.

M4 temporarily projects metadata-only compatibility responses so `just demo`
runs on the core engine stack. M4.1 defines enough client/server contract for
node details, runtime-owned buffers, and render products that the client no
longer sits at `(Waiting for state data...)`.

## Context

M3.2 adds `RuntimeBufferStore` and `RuntimeProduct::Buffer` so texture pixels,
fixture colors, and output channel bytes have engine-owned identity. M4 ports
the old MVP nodes as first-class core nodes onto that model.

Once real core nodes are publishing and consuming store-backed buffers/products,
sync should stop treating frame-heavy bytes as ordinary node-state fields. The
sync layer needs explicit refs, metadata, versions, and update policy.

## In scope

- Define the wire/view representation for runtime buffer references.
- Define how render-product references surface in node details where clients
  need to display or inspect them.
- Decide how buffer metadata and versions are exposed to clients.
- Add client-side buffer cache behavior for initial sync and incremental
  updates.
- Restore the client-visible `node_details` needed by the current demo/view
  path, especially texture/shader/fixture/output status and fixture runtime
  state.
- Replace or narrow legacy heavy `NodeState` snapshots where the client can use
  buffer refs instead.
- Preserve compatibility snapshots where still needed for old clients or tests.
- Define cleanup semantics for buffers that disappear when nodes are destroyed
  or reinitialized.
- Add focused sync/view tests for create, update, resize, and removal cases.

## Out of scope

- Porting runtime nodes; that is M4.
- Making the core engine the default runtime; that is M5.
- Compression, scaling, binary chunking, or adaptive throttling unless needed to
  establish the minimal contract.
- GPU-backed transport or desktop-only resource handles.
- Replacing every legacy state field unrelated to heavy runtime buffers.
- Source reload, node deletion, output close/unregister, and other lifecycle
  parity; M4.2 owns those.

## Key decisions

- **Runtime pressure first:** M4 should reveal the real buffer domains and update
  cadence before sync commits to a final shape.
- **Before cutover:** buffer-aware sync should land before M5 so the cutover does
  not depend on legacy full-copy state for frame-heavy data.
- **Refs plus compatibility:** introduce explicit buffer refs while keeping
  compatibility snapshots only at boundaries that still need them.
- **Detail parity first:** restore the visible client contract before deleting
  the old runtime path.

## Success criteria

- Clients can identify and cache runtime buffers by stable IDs/refs.
- Buffer metadata/version changes are visible through sync.
- Incremental sync can update buffer-aware views without treating heavy bytes as
  ordinary scalar/node-state payloads.
- `just demo` no longer leaves loaded nodes at `(Waiting for state data...)`.
- Node destruction or incompatible reinit removes or invalidates client-visible
  buffer refs.
- M5 can cut over without designing buffer sync from scratch.
