# Milestone 3.1: State wire projection

## Goal

Harden the sync projection path before porting legacy node runtime behavior onto
the core engine.

This milestone should make it clear how, given a client frame id and watch/detail
interests, the runtime projects current state into a client-usable form. It also
separates client-visible compatibility snapshots from runtime-owned data.

## Context

M3 moved authored legacy node config to `node.toml` and source-owned loading.
M4 will move legacy shader / fixture / output behavior onto the core engine. The
gap between those milestones is the sync layer: existing legacy state currently
sends texture bytes, fixture lamp colors, output bytes, config details, and
partial state snapshots through `ProjectResponse::GetChanges`.

Before M4 depends on that path, the repo needs a tighter `SyncProjection`
contract and tests around the current serialization/client-view behavior.

## In scope

- Review and harden legacy `GetChanges` serialization/deserialization.
- Fix incomplete client-view application of config and state details if needed.
- Define vocabulary for sync projections versus runtime products/buffers.
- Inventory heavy state fields that should become store-backed products/buffers
  in M3.2.
- Add focused tests for partial state merge, config detail application, and
  supported wire round trips.

## Out of scope

- Implementing the full runtime buffer/product store.
- Porting legacy runtime behavior to the core engine.
- Retiring `LegacyProjectRuntime`.
- Replacing the legacy `GetChanges` protocol.
- Designing compression, scaling, throttling, binary chunks, or texture diff
  transport policy.

## Key decisions to make during planning

- **Projection name:** use `SyncProjection` for the conceptual boundary and
  `LegacySyncProjection` for the current `GetChanges` compatibility path.
- **Compatibility boundary:** which legacy `NodeState` fields remain wire
  snapshots for M4, and which should become store-backed runtime data later.
- **Client config handling:** whether `ProjectView` should store a concrete
  config enum/helper instead of placeholder `Box<dyn NodeConfig>` values.
- **Round-trip support:** whether `SerializableProjectResponse` should fully
  deserialize from its custom serialized shape or only support server-to-client
  encoding plus client apply tests.

## Success criteria

- The M4 port has a documented projection boundary for legacy state/config sync.
- Client-view config and partial-state behavior is correct enough for parity
  tests.
- Serialization tests cover the supported legacy `GetChanges` path.
- M3.2 has a concrete inventory of texture/color/raw buffer requirements.
