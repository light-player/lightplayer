# Milestone 4.5: Core node state model

## Goal

Design the long-term representation for node state/details in the new core node
system after M4.1-M4.4 restore parity.

## Context

M4.1 may extend legacy-compatible state shapes to carry semantic resource refs
so sync can work without redesigning every node-state concept at once. Those
compatibility additions should be named and organized so they are easy to find
and replace.

The durable question remains: what is node state in the core runtime? It may be
a typed core detail model, value/resource field projections, explicit debug
state, compatibility adapters, or some combination.

## In scope

- Decide whether `NodeState` remains legacy-only or becomes a core concept.
- Define how semantic state fields reference value, buffer, and render-product
  domains.
- Decide how `RuntimeStateAccess` should evolve beyond a marker trait.
- Separate user-visible node details from debug/runtime internals.
- Retire or quarantine legacy compatibility state fields added during M4.1.
- Update wire/view architecture to match the chosen model.

## Out of scope

- Restoring immediate demo detail sync; M4.1 owns that.
- Source reload/lifecycle parity; M4.2 owns that.
- Runtime behavior parity; M4.3 owns that.
- Legacy runtime removal; M5 owns that.

## Success criteria

- Core nodes have a clear state/detail projection contract.
- Compatibility state adapters are isolated and easy to remove.
- Resource refs preserve semantic field names without tying core design to old
  heavy snapshot structs.
- Future nodes can expose state/details without copying legacy runtime patterns.
